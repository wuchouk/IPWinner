## 一、產品願景

### 問題陳述

IP Winner（智財事務所）每日處理約 100 封業務郵件，涵蓋專利、商標案件的客戶往來、代理人通訊、政府公文。目前這些信件的分類、命名、歸檔完全靠人工，需要 **1.15 個全職人力**（每日 9.2 小時）。

核心痛點：
1. **時間成本高**：每封信平均 5.5 分鐘（閱讀→判斷→命名→歸檔），每月 202 小時
2. **知識門檻高**：新人需 1-2 個月熟悉案號格式、收發碼邏輯、命名慣例
3. **品質不穩定**：人工疲勞導致 2-5% 錯誤率，且難以系統性改善
4. **單點風險**：負責人請假或離職，歸檔工作立刻中斷

### 產品定位

一套部署在 Google Apps Script 上的**智慧郵件分類與歸檔系統**，結合規則引擎（98.8% 準確率）與 LLM 語義理解（處理 1.2% 邊界案例），實現郵件的自動分類、智慧命名、自動歸檔，並透過人機回饋持續學習。

### 核心價值主張

| 指標 | 人工 | AI 系統 |
|------|------|---------|
| 每封處理時間 | 5.5 分鐘 | ~2 秒 |
| 月成本 | NT$40,000-50,000 | **NT$160**（Gemini 3.0 Flash） |
| 錯誤率 | 2-5% | 1.2%（持續降低） |
| 可用性 | 工作日 8 小時 | 24/7 |
| 知識傳承 | 依賴個人 | 系統化規則 + 學習紀錄 |

**年度節省：NT$478,000-658,000（99.6% 成本降低）**

## 二、使用者角色

### 主要使用者：事務所行政人員
- **日常操作**：檢查 Gmail 中的 `AI/待確認` 和 `AI/未知來源` 標籤（每天 5-10 分鐘）
- **回饋修正**：在 Gmail 修改收發碼標籤、在 Sheet 修正語義名稱
- **期望**：系統自動處理 95%+ 郵件，只需關注例外情況

### 次要使用者：事務所管理者
- **監控品質**：查看處理統計、正確率趨勢
- **管理規則**：維護 Sender 名單、審核學習紀錄
- **決策**：根據 correction log 判斷是否需要調整分類規則

### 間接受益者：專利/商標承辦人
- **使用歸檔成果**：在 Drive 案號資料夾中查找 EML 和附件
- **命名品質**：語義化檔名讓搜尋更直覺（如 `20260314-FA-BRIT21002PUS5-送件報告-(ROA1).eml`）

## 三、功能清單

### Phase 1（已完成）

#### 3.1 自動收發碼判定
- 8 種收發碼：FC/TC/FA/TA/FG/TG/FX/TX
- F（收到）/T（寄出）× C（客戶）/A（代理人）/G（政府）/X（未知）
- Sender 名單比對（domain 或完整 email）
- 規則引擎 baseline 準確率：98.8%

#### 3.2 智慧案號提取
- 正規表示式匹配：`[A-Z0-9]{4}\d{5}[PMDTABCW][A-Z]{2}\d*`
- 支援變體：完整案號、基礎案號、僅客戶碼、多案號（～/等/、）
- 主旨案號 vs 內文案號區分（只有主旨案號觸發歸檔）
- LLM 判斷 filing_case_numbers（決定實際歸檔對象）

#### 3.3 LLM 語義命名
- 前綴引導 + 自由摘要（非固定模板匹配）
- 651 個正規化模板中，80 個高頻模板（47.2%）+ 571 個自由描述（52.8%）
- 繁體中文語義名，25 字以內
- 括號事項碼：案件類型、OA 序號、年費/領證、帳單事項
- 截止日強制納入（行動期限優先於背景日期）

#### 3.4 案件類別判定
- 專利：案號含 P（發明）/M（新型）/D（設計）
- 商標：案號含 T
- 未分類：無案號、無類型碼、或混合

#### 3.5 自動歸檔
- Drive 資料夾結構：`{專案根}/專利|商標|未分類/{案號}/`
- EML 檔名：`{日期}-{收發碼}-{案號標記}-{語義名}.eml`
- 附件：原始檔名保留（Phase 1 不重命名）
- 多案號：每個案號資料夾各存一份實體 EML
- 寄出信（TA/TC/TG）：只存 EML，不存附件
- 重跑保護：同檔名 + 同大小（±100 bytes）自動跳過

#### 3.6 Gmail 標籤系統
- 收發碼標籤（互斥 1 個）：AI/FC ~ AI/TX
- 案件類別標籤（互斥 1 個）：AI/專利、AI/商標、AI/未分類
- 案號狀態標籤（0-1 個）：AI/多案號、AI/無案號
- 問題標籤（0-多個）：AI/待確認、AI/自動辨識來源、AI/未知來源、AI/已跳過、AI/附件下載錯誤、AI/處理失敗

#### 3.7 未知來源三階判定
1. LLM 從內文推斷角色 + 信心 ≥ 0.6 → 具體碼 + `AI/自動辨識來源`（待人確認）
2. LLM 推斷但信心 < 0.6 → 保留 FX/TX + `AI/未知來源`
3. 完全無法判斷 → `AI/已跳過`

#### 3.8 三通道回饋學習
- **通道 1：Gmail 標籤修正** — 移除/修改收發碼標籤 → 更新 Sender 名單
- **通道 2：Sheet 修正名稱** — 填入偏好語義名 → Drive 檔案自動改名
- **通道 3：Sheet 直填收發碼** — 直接指定最終收發碼 → 更新 Sender 名單
- 修正紀錄 → Few-shot 注入 LLM prompt（最近 20 筆）
- 每週一自動整理學習紀錄（LLM 歸納 → Google Doc）

#### 3.9 批次處理與排程
- 20 封/批寫入 Google Sheet
- 25 分鐘 timeout 防護，自動排下一個 trigger 接續
- Checkpoint 機制：記錄最後處理的 messageId，支援斷點續跑
- 每日排程（早上 7-8 點）或手動觸發

#### 3.10 操作介面
- Google Sheet 自訂選單（📧 Email Processor V3）
- 試跑模式：50 封 / 10 封 / 單封
- 處理統計、排程管理、API Key 設定
- LLM Prompt 匯出與學習紀錄整理

### Phase 2（規劃中）
- [ ] 附件重新命名（D/A/F 版本管理）
- [ ] 精確子類型碼自動判斷
- [ ] 附件合併/拆分
- [ ] 完全自動化（移除人工確認步驟）

### 未來願景：SaaS 產品化
- 多租戶支援（每客戶獨立 Sheet + Drive + 規則）
- 案號格式可設定化
- Prompt 模板化（前綴、代理人代碼可自訂）
- Cloud Functions 架構（突破 Apps Script 6 分鐘限制）

## 五、技術約束

| 約束 | 說明 |
|------|------|
| Google Workspace 帳號 | Apps Script 執行上限 30 分鐘 |
| 單檔部署 | 方便客戶「貼上即用」，但限制模組化 |
| Gmail thread model | Label 加在 thread 上，需用 messageId 去重 |
| Apps Script 單線程 | Drive 操作無法並行，LLM 呼叫用 fetchAll 並行 |
| Gemini API | 偶爾截斷 JSON 回應，需 _repairJson() 修復 |
