## 一、核心概念

### 收發文碼（8 種）

| 碼 | 方向 | 含義 | 判斷依據 |
|----|------|------|----------|
| FC | 收 | 客戶→我們 | sender = 客戶 domain |
| TC | 發 | 我們→客戶 | sender = ipwinner.com, recipient = 客戶 |
| FA | 收 | 代理人→我們 | sender = 代理人 domain |
| TA | 發 | 我們→代理人 | sender = ipwinner.com, recipient = 代理人 |
| FG | 收 | 官方→我們 | sender = tipo.gov.tw 等政府 domain |
| TG | 發 | 我們→台灣官方 | sender = ipwinner.com, recipient = 政府 |
| FX | 收 | 未知來源→我們 | sender 不在名單內 |
| TX | 發 | 我們→未知對象 | sender = ipwinner.com, recipient 不在名單內 |

### 案號格式

完整格式：`[A-Z0-9]{4}\d{5}[PMDTABCW][A-Z]{2}\d*`
範例：`BRIT21002PCN`, `BRIT24001DUS1`, `KOIT20004TCN7`

實際使用中有幾種變體：
- **完整案號**（1,076 封 / 86.1%）：BRIT21002PUS5
- **基礎案號無國碼**（27 封 / 2.2%）：BRIT18001（案件初期尚未確定國別）
- **僅客戶代碼**（58 封 / 4.6%）：BRIT（跨多國多案）
- **多案號 ～**（72 封 / 5.8%）：BRIT24001～24008（全形波浪號）
- **多案號 等**（10 封 / 0.8%）：BRIT18001TCN等
- **多案號 、**（2 封 / 0.2%）：BRIT24001DJP1、BRIT24002DJP1
- **代理人名稱**（4 封 / 0.3%）：K&K（極少數跨案溝通）

### 案件類別判斷

根據案號中的類型碼判斷：
- **專利**：P（發明）、M（新型）、D（設計）
- **商標**：T
- **未分類**：無案號、或案號中無類型碼、或混合專利+商標

## 二、Gmail 標籤系統

### 設計原則
- 沒有問題的信 = 只有分類標籤（收發碼 + 案件類別），代表已處理
- 有問題的信 = 額外掛問題標籤，需要人關注

### 問題標籤（有問題才掛，0 或多個）

| 標籤 | 觸發條件 | eml 是否已下載 | 人的操作 | 系統行為 |
|------|---------|--------------|---------|---------|
| AI/待確認 | LLM 信心 < 0.8，或無法完整分類 | ✅（LLM 推測命名或原始標題） | 確認OK→移除標籤；有錯→改收發碼/類別標籤 | 偵測標籤變更→記錄修正→更新檔名 |
| AI/自動辨識來源 | sender 不在名單，但 LLM 從內文推斷出角色（C/A/G） | ✅（LLM 推測命名，收發碼用推斷結果如 FC/FA/FG） | 確認推斷正確→移除此標籤；有誤→改收發碼標籤 | 偵測標籤移除→確認 sender 角色→自動加入 Sheet 1 名單 |
| AI/未知來源 | sender 不在名單 + LLM 也無法從內文判斷角色 | ✅（用 FX/TX，LLM 推測命名） | 改 FX→正確碼（如 FA），或更新 Sheet 名單 | 偵測標籤變更→加入名單→重新分類 |
| AI/已跳過 | sender 未知 + 內容完全無法辨識 | ❌ | Sheet 標記要處理，或不管 | 下次掃描檢查 Sheet |
| AI/附件下載錯誤 | eml 成功但附件下載失敗 | eml ✅，附件 ❌ | 不用管 | 自動重試，3 次後停止 |
| AI/處理失敗 | 系統錯誤（API/LLM/腳本） | 可能都沒有 | 不用管 | 自動重試，3 次後停止 |

### 未知來源的三階判定流程

```
sender 不在 Sheet 1 名單
    │
    ▼
[LLM 從內文推斷角色]
    ├── 有信心（0.6+）→ 採用推斷碼（如 FC/FA/FG）
    │     └── 掛 AI/自動辨識來源 + AI/FC（或 FA/FG）
    │
    ├── 低信心（< 0.6）→ 保留 FX/TX
    │     └── 掛 AI/未知來源 + AI/FX（或 TX）
    │
    └── 完全無法判斷 → AI/已跳過
```

**關鍵差異：`AI/自動辨識來源` vs `AI/未知來源`**

- `AI/自動辨識來源`：LLM 認為它知道是誰，但 sender 不在名單裡所以要人確認。收發碼已經是具體的（FC/FA/FG），人只需要看一眼確認就好。確認後系統自動把 sender 加入名單。
- `AI/未知來源`：LLM 也搞不清楚，收發碼是 FX/TX，需要人主動判斷角色。

這個設計讓「需要人工介入」的量大幅減少——大部分 FX/TX 會變成 `AI/自動辨識來源`，人只要快速掃一眼確認；只有少數真正無法判別的才需要深入看。

### 收發碼標籤（互斥，1 個）

`AI/FC`, `AI/TC`, `AI/FA`, `AI/TA`, `AI/FG`, `AI/TG`, `AI/FX`, `AI/TX`

### 案件類別標籤（互斥，1 個）

`AI/專利`, `AI/商標`, `AI/未分類`

### 案號狀態標籤（可選，0 或 1 個）

`AI/多案號`, `AI/無案號`

### 標籤組合範例

| 情境 | 標籤 |
|------|------|
| 正常處理：代理人來的專利送件報告 | AI/FA, AI/專利 |
| 正常處理：多案號商標通知 | AI/TC, AI/商標, AI/多案號 |
| 待確認：LLM 不太確定分類 | AI/待確認, AI/FA, AI/專利 |
| 自動辨識來源：sender 未知但 LLM 從內文判斷是代理人 | AI/自動辨識來源, AI/FA, AI/專利 |
| 未知來源：sender 未知且 LLM 也無法判斷 | AI/未知來源, AI/FX, AI/專利 |
| 已跳過：完全不認識 | AI/已跳過 |
| 附件錯誤：eml 有但附件失敗 | AI/附件下載錯誤, AI/FA, AI/專利 |

**人工覆核流程（`AI/自動辨識來源`）：**

1. 在 Gmail 篩選 `AI/自動辨識來源` → 看到待確認的信件清單
2. 打開信件 → 確認 LLM 推斷的角色是否正確（收發碼標籤已標示如 AI/FC）
3. 正確 → 移除 `AI/自動辨識來源` 標籤 → 系統偵測到移除 → 自動將 sender 加入 Sheet 1 名單
4. 錯誤 → 改收發碼標籤（如 AI/FC → AI/FA）+ 移除 `AI/自動辨識來源` → 系統記錄修正並用正確角色加入名單
5. Sheet 2 處理紀錄同步更新

## 三、Ground Truth 分析結果

## 六、LLM 分類 System Prompt

以下是給 Gemini 3.0 Flash 的分類 prompt。

### 設計原則
- 規則引擎做前處理（抽案號、判 F/T、查 sender 角色）
- LLM 做語義分類 + 產生檔名
- 策略是「**前綴引導 + 自由摘要**」而不是固定規則匹配
- 每次呼叫時，動態注入最近的人工修正紀錄作為 few-shot examples

---

```
你是 IP Winner 智財事務所的 email 分類助手。根據以下資訊判斷 email 類型並產生語義檔名。

## 輸入格式
你會收到一封 email 的結構化資訊：
- subject: 信件標題（已去除 RE:/Fwd: 前綴）
- original_subject: 原始標題（含 RE:/Fwd:）
- direction: "F"（收到）或 "T"（寄出）
- role: "C"（客戶）, "A"（代理人）, "G"（政府）, "X"（未知）
- sender: 寄件人 email
- recipients: 收件人列表
- case_numbers: 從標題/內文/附件名抽出的案號列表
- has_attachments: 是否有附件
- attachment_names: 附件檔名列表
- body_snippet: 信件內文前 500 字

## 第一步：確認收發文碼
規則引擎已根據 direction + role 預判了收發碼，你需要驗證：
- F + C → FC | F + A → FA | F + G → FG | F + X → FX
- T + C → TC | T + A → TA | T + G → TG | T + X → TX

### 重要：未知來源角色推斷
當 role = "X"（sender 不在名單）時，你必須嘗試從信件內文推斷角色：

**代理人（A）線索：**
- 送件報告、Filing Report、確認承辦
- 帳單、Invoice、Debit Note
- 信件使用正式商業英文，提及 filing、prosecution、registration
- 附件名稱含代理人相關命名

**客戶（C）線索：**
- 簡短回覆（ok、確認、同意）
- 指示答辯方向、領證指示
- 信件中提及「我們的產品」、「我司」
- sender domain 為一般企業（非法律事務所）

**政府（G）線索：**
- 官方通知、核駁、電子收據
- sender domain 含 gov 或官方機構名

如果你能推斷角色，在 `inferred_role` 欄位填入推斷的角色（C/A/G），並將收發碼改為具體碼（如 FX→FA）。

## 第三步：判斷案件類別
- 案號含 P/M/D → 專利

## 輸出格式（JSON）

**一般情況（sender 角色已知）：**
```json
{

  "eml_filename": "送件報告-(ROA1)",
  "case_category": "專利",

}
```

**未知來源但 LLM 成功推斷角色：**
```json
{
  "send_receive_code": "FA",
  "eml_filename": "送件報告-(OA1)",
  "case_category": "專利",
  "case_status": null,
  "inferred_role": "A",
  "confidence": 0.75,
  "reasoning": "sender newagent@lawfirm.co.jp 不在名單，但內文為 Filing Report 格式，附件含申請書掃描，判斷為代理人(A)。建議確認後加入名單。"
}
```

**未知來源且 LLM 無法推斷：**
```json
{
  "send_receive_code": "FX",
  "eml_filename": "詢問專利狀態",
  "case_category": "專利",
  "case_status": null,
  "inferred_role": null,
  "confidence": 0.4,
  "reasoning": "sender unknown@company.com 不在名單，內文簡短無法判斷角色，保留 FX。"
}
```

### inferred_role 說明
- `null`：sender 已在名單中（角色已知），或 sender 未知且 LLM 無法推斷
- `"C"` / `"A"` / `"G"`：LLM 從內文推斷的角色
- 當 `inferred_role` 不為 null 時，系統加掛 `AI/自動辨識來源` 標籤

### confidence 說明
- 0.8+ : 自動處理
- 0.6-0.8 : 角色已知→自動處理；角色推斷(inferred_role)→加掛 AI/自動辨識來源
- 0.5-0.6 : 標記 AI/待確認（推斷的角色信心不足，保留 FX/TX）
- 0.5以下 : 標記 AI/待確認（使用原始標題命名）

### 注意事項
1. RE:/Fwd:/回覆:/轉寄: 前綴先去除再判斷，但保留原始標題供參考
2. 語義名稱必須用繁體中文，不要照抄英文標題
3. 語義名稱控制在 25 字以內（不含括號事項碼）
4. OA/ROA 的序號要從標題或內文中提取
5. 如果完全無法分類，eml_filename 設為原始標題的中文摘要
6. 帳單的代理人代碼要從 sender domain 對應
7. 當 role=X 時，務必嘗試從內文推斷角色，只有真正無法判斷才保留 X
```

## 七、分類流程

```
收到 email
    │
    ▼
[前處理層 - 規則引擎]
    ├── 抽取案號（regex + 內文搜尋）
    ├── 判斷 F/T（sender = ipwinner.com?）
    ├── 查 sender role（Google Sheet Sender 名單）→ C/A/G/X
    ├── 去除 RE:/Fwd: 前綴
    └── 抽取附件名稱列表
    │
    ▼
[分類層 - Gemini 3.0 Flash]
    ├── 輸入：結構化 email 資訊 + 高頻模板 + 近期修正
    ├── 輸出：JSON（收發碼、語義檔名、類別、信心、inferred_role）
    │
    ├── confidence ≥ 0.8 → 正常處理
    ├── confidence 0.5-0.8 → 加掛 AI/待確認
    └── confidence < 0.5 → 加掛 AI/待確認，用原始標題命名
    │
    ▼
[來源辨識層]（僅 sender role = X 時觸發）
    ├── LLM 從內文/主旨/附件推斷 sender 角色
    │     ├── inferred_role 有值 + 信心 ≥ 0.6
    │     │     → 收發碼改為具體碼（FX→FC/FA/FG, TX→TC/TA/TG）
    │     │     → 加掛 AI/自動辨識來源
    │     │
    │     ├── inferred_role 有值但信心 < 0.6
    │     │     → 保留 FX/TX
    │     │     → 加掛 AI/未知來源
    │     │
    │     └── 完全無法判斷
    │           → AI/已跳過
    │
    ▼
[命名層]
    ├── eml: yyyymmdd-收發碼-案號-語義名稱.eml
    └── 附件: 原始檔名（Phase 1 不重新命名）
    │
    ▼
[存檔層]
    ├── 下載 eml + 附件（所有收發碼都下載）
    ├── 多案號 → 複製到每個案號資料夾
    └── 加 Gmail 標籤
    │
    ▼
[回授層 - 每次掃描]
    ├── 檢查 AI/待確認 標籤是否被人移除或修改
    ├── 檢查 AI/自動辨識來源 是否被人移除（=確認正確）→ 加入 Sheet 1 名單
    ├── 檢查 AI/自動辨識來源 的收發碼是否被人修改（=推斷錯誤）→ 用修正角色加入名單
    ├── 檢查 AI/未知來源 標籤是否被人改成具體收發碼
    ├── 檢查 Google Sheet Sender 名單是否有新增
    ├── 記錄修正到 Sheet 處理紀錄
    └── AI/附件下載錯誤、AI/處理失敗 → 自動重試（上限 3 次）
```

## 八、Google Sheet 架構

### Sheet 1：Sender 名單
| 欄位 | 說明 |
|------|------|
| Email 或 Domain | 如 @bskb.com 或 tanaka@gmail.com |
| 角色（C/A/G） | 客戶/代理人/政府 |
| 名稱備註 | 如 BSKB - 美國代理人 |

查詢邏輯：email 精確匹配優先 → domain 匹配。

### Sheet 2：處理紀錄
| 欄位 | 說明 |
|------|------|
| messageId | Gmail message ID |
| 日期 | 處理日期 |
| 原始標題 | 信件標題 |
| sender | 寄件人 email |
| AI收發碼 | LLM 判定的收發碼（含推斷後的具體碼） |
| AI推斷角色 | LLM 推斷的 sender 角色（C/A/G/null）。有值=sender 不在名單但 LLM 推斷了角色 |
| AI案號 | 提取的案號 |
| AI語義名 | LLM 產生的語義檔名 |
| AI信心 | confidence score |
| AI案件類別 | 專利/商標/未分類 |
| 來源確認狀態 | pending（待確認）/ confirmed（已確認）/ corrected（已修正）/ na（角色已知） |
| 最終收發碼 | 人工修正後的收發碼（null=未修正） |
| 最終語義名 | 人工修正後的語義名（null=未修正） |
| 客戶修改名稱 | 客戶偏好的命名方式（手動填寫）。系統會自動收集此欄有值的紀錄，彙整成 few-shot examples 注入 LLM prompt，讓命名風格逐漸貼近客戶習慣 |
| 修正時間 | 人工修正的時間戳 |
| 修正來源 | tag_change / sheet_edit |
| 重試次數 | 附件下載或處理失敗的重試次數 |

此表是 **correction log**，LLM 學習的來源。

**客戶修改名稱的回授流程：**
1. 客戶在「處理紀錄」Sheet 的「客戶修改名稱」欄填入偏好的語義名稱
2. 下次系統執行時，自動收集有填值的紀錄，產生 AI/客戶 配對範例
3. 範例注入 LLM prompt 的「近期人工修正紀錄」區塊
4. LLM 根據範例學習客戶命名風格（如加入截止日提示、特定前綴偏好等）

**來源確認狀態欄位的用途：**
- `na`：sender 已在 Sheet 1 名單中，無需確認
- `pending`：LLM 推斷了角色（`AI/自動辨識來源`），等待人工確認
- `confirmed`：人確認 LLM 推斷正確，已移除 `AI/自動辨識來源` 標籤，sender 已自動加入 Sheet 1
- `corrected`：人修正了 LLM 推斷的角色，sender 以修正後的角色加入 Sheet 1

### Sheet 3：分類規則擴充
| 欄位 | 說明 |
|------|------|
| 規則ID | 如 TPL-001, NEW-001 |
| 觸發條件 | 標題關鍵字/sender 特徵/內文特徵 |
| 收發碼 | 對應的收發碼 |
| 語義檔名模板 | 如 送件報告-(XX) |
| 來源 | ground_truth / excel / correction |
| 新增日期 | 規則建立日期 |

初始內容從 ground truth 80 個高頻模板 + Excel 108 條建立。人工修正的新模式會被加入。

### Sheet 4：設定
| 參數 | 值 |
|------|-----|
| 信心閾值 | 0.8 |
| 附件重試上限 | 3 |
| 掃描間隔(分鐘) | 15 |
| batch 大小 | 20 |
| 最後處理的 messageId | （自動更新） |
| 本次已處理數量 | （自動更新） |

## 九、回授學習機制

### 短期（自動）
每次呼叫 LLM 時，從 Sheet 2 的 correction log 抽取最近 20 筆修正，動態注入 prompt 的 few-shot examples。

### 長期（半自動）
每月分析 correction log 的高頻錯誤模式，整理成正式規則加入 Sheet 3。

### 雙向同步：未知來源

**路徑 A — AI 自動辨識來源（最常見）：**
1. LLM 推斷 sender 角色 → 掛 `AI/自動辨識來源` + 具體收發碼（如 AI/FA）
2. 人確認正確 → 移除 `AI/自動辨識來源` 標籤
3. 系統偵測標籤移除 → 自動將 sender email/domain 加入 Sheet 1 名單（角色=LLM推斷值）
4. 該 sender 的後續信件不再需要人工確認

**路徑 B — AI 推斷錯誤：**
1. 人看到 `AI/自動辨識來源` + AI/FA → 發現其實是客戶
2. 人改標籤 AI/FA → AI/FC + 移除 `AI/自動辨識來源`
3. 系統偵測 → 用修正後的角色（C）加入 Sheet 1 名單 → 更新檔名

**路徑 C — 完全未知（LLM 也判斷不了）：**
- 人改 Gmail 標籤（AI/FX → AI/FA）→ 系統偵測 → 自動更新 Sheet 1 名單
- 或：人更新 Sheet 1 名單 → 系統重新處理所有 `AI/未知來源` 信件

## 十、批次執行與 Timeout 防護

### 執行策略
- Workspace 帳號上限 30 分鐘
- 每 20 封一個 batch 寫入 Google Sheet
- 每個 batch 完成後更新 checkpoint（Sheet 4）
- 快到 25 分鐘時自動排下一個 trigger 接續

### Checkpoint 機制
- 每次寫入 Sheet 時記錄 lastProcessedMessageId
- 下次啟動從斷點續跑，不重複處理
